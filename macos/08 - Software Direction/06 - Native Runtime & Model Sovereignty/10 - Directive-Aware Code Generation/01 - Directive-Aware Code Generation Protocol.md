## PURPOSE

Define how ICOS agents generate code strictly based on directives, ensuring outputs are fully aligned with system doctrine and architecture.

---

## CORE POSITION

Code is derived from directives.

Generation without directive grounding is invalid.

---

## GENERATION PIPELINE

Every code generation must follow:

1. directive resolution
2. context aggregation
3. constraint extraction
4. architecture mapping
5. code generation
6. validation

No step may be skipped.

---

## PIPELINE ENFORCEMENT

The pipeline must be enforced by system logic:

- no direct code generation bypass
- no skipped stages
- each stage must log completion

Pipeline integrity must be verifiable.

---
---

## DIRECTIVE RESOLUTION

The agent must identify and read:

- Product Identity
- Core Directives
- Relevant Standards/Architecture
- Target file(s)

Directives define the boundaries of code.

---

## DIRECTIVE PRIORITY ORDER

Directives must be resolved in strict order:

1. Product Identity
2. Core Directives
3. Architecture & Standards
4. Local File Context

Higher-order directives override lower-order inputs.

---
---

## CONTEXT AGGREGATION

The agent must combine:

- global doctrine
- system architecture
- workspace files
- current task scope

Context must be complete before generation.

---

## CONTEXT COMPLETENESS CHECK

The agent must verify:

- all required documents are loaded
- no missing dependencies
- target files are accessible

If incomplete, generation must halt.

---
---

## CONSTRAINT EXTRACTION

From directives, the agent must extract:

- structural rules
- naming conventions
- architectural patterns
- safety constraints

These constraints govern code output.

---

## CONSTRAINT LOCKING

Extracted constraints must be:

- treated as immutable during generation
- enforced across all output

No deviation allowed.

---
---

## ARCHITECTURE MAPPING

The agent must map directives to:

- modules
- services
- file structure
- runtime behavior

Code must reflect actual system design.

---

## STRUCTURAL CONSISTENCY

Mapping must:

- match existing system modules
- respect folder hierarchy
- preserve naming conventions

No structural invention is allowed.

---
---

## CODE GENERATION

Generated code must:

- be executable
- be complete
- integrate with existing system
- follow naming and structure rules

No placeholder or pseudo-code.

---

## EXECUTION VALIDITY

Generated code must:

- compile or run (where applicable)
- align with runtime interfaces
- avoid dead or unreachable logic

---
---

## VALIDATION

Before finalizing, the agent must verify:

- directive alignment
- structural correctness
- integration compatibility
- absence of contradictions

---

## VALIDATION ENFORCEMENT

Validation must include:

- directive compliance check
- structural integrity check
- integration compatibility check

Failure in any check must block output.

---
---

## FILE INTEGRATION

Generated code must:

- be written into correct files
- update existing structures
- preserve metadata and formatting

No isolated code blocks.

---

## FILE OWNERSHIP VERIFICATION

Before writing, the agent must:

- verify correct file ownership
- confirm correct insertion location

Incorrect targeting is prohibited.

---
---

## NO GUESSING RULE

The agent must not:

- invent architecture
- assume missing structure

If context is missing, request scan or file.

---

## NO OVERLAY RULE

Code must:

- modify root logic
- not introduce parallel layers

---

## SINGLE-SOURCE CORRECTION

All fixes must:

- modify the canonical source
- not introduce duplicate logic paths

---
---

## CONSISTENCY RULE

Code must be consistent with:

- previous implementations
- global patterns
- shared tokens and structures

---

## ERROR HANDLING

If directive conflict occurs:

- stop generation
- surface conflict
- request resolution

---

## CONFLICT RESOLUTION PROTOCOL

On conflict:

- identify conflicting directives
- surface conflict clearly
- require user or higher-order resolution

No silent compromise.

---
---

## MULTI-AGENT ALIGNMENT

All agents must:

- follow same generation protocol
- produce compatible code

---

## FAILURE CONDITION

Code is generated without directive grounding or violates system architecture.


---

## GENERATION TRACEABILITY

Each generation must record:

- directives used
- files referenced
- constraints applied
- validation results

Trace must be auditable.

---

## SUCCESS CONDITION

All code is deterministically derived from directives and integrates seamlessly into the ICOS system.
