## PURPOSE

Define strict conduct rules for agent coding and response behavior to ensure deterministic, doctrine-aligned outputs and direct file-grounded execution.

---

## CORE POSITION

All outputs must align with directives.

All changes must affect source files.

---

## DIRECTIVE-FIRST RULE

Before any code or response, the agent must:

1. read Product Identity
2. read Core Directives
3. read relevant Architecture/Standards
4. read target local files

No generation without directive grounding.

---

## EXECUTION PRECONDITIONS

Before execution, the agent must:

- verify target file is open
- verify ownership chain (layout, component, script)
- confirm path validity

No edit without verified context.

---

## RESPONSE ALIGNMENT

Responses must be:

- precise
- structured
- minimal
- non-theatrical
- non-generic

No motivational or assistant-style tone.

---

## CODE GENERATION RULE

Code must:

- map directly to directives
- be executable
- avoid placeholders
- reflect actual system architecture

No speculative code.

---

## FILE-BOUND EXECUTION

All meaningful output must:

- read from files
- write to files
- update canonical sources

No chat-only solutions.

---

## FILE OPEN REQUIREMENT

The agent must:

- request exact file path
- wait for confirmation
- then apply changes in-place

No assumed open state.

---

## NO OVERLAY (NOL)

The agent must:

- fix at root source
- avoid patches, wrappers, or parallel logic

Overlaying is prohibited.

---

## ROOT-CAUSE DECLARATION

For each fix, the agent must:

- identify root cause in one sentence
- apply single-source correction

Multiple-layer fixes are prohibited.

---

## SAME MEANS SAME

When duplicating behavior or structure:

- use identical source logic
- mirror tokens, spacing, ownership

No approximation.

---

## PATH DISCIPLINE

The agent must:

- use verified absolute paths
- never guess directories

If unknown, request scan.

---

## NO REOPEN LOOP

The agent must:

- avoid repeated open requests for same file
- batch related edits in one pass

Reopen loops are prohibited.

---

## CHANGE DISCIPLINE

Edits must be:

- atomic
- complete
- non-fragmented

If direct edit fails → provide full file replacement.

---

## VERIFICATION STEP

After each change, the agent must:

- state whether fix is root-level
- confirm no overlay introduced

Unverified fixes must be flagged.

---

## METADATA INTEGRITY

The agent must:

- preserve YAML order
- preserve spacing and sections
- never remove titles (e.g., Change Log)

---

## TOKEN CONSISTENCY

The agent must:

- use shared tokens from global system
- avoid local hardcoding

All style/logic must derive from canonical sources.

---

## SAFETY & PERMISSIONS

The agent must:

- respect file permissions
- avoid exposing restricted data
- require approval for high-impact changes

---

## ERROR HANDLING

On uncertainty:

- pause
- request context/scan

No blind execution.

---

## CONSISTENCY ACROSS AGENTS

All agents must:

- follow identical standards
- produce consistent outputs
- bind to same doctrine set

---

## TRACEABILITY

All actions must record:

- file path
- change type
- execution context

Trace must support audit and rollback.

---

## FAILURE CONDITION

Agent produces ungrounded responses, uses overlays, guesses paths, or avoids file updates.

---

## SUCCESS CONDITION

Agent produces directive-aligned outputs and performs deterministic, source-level changes with full system consistency.
