## PURPOSE

Define the canonical software direction for ICOS as a cognitive operating system and bind all implementation to product definition, doctrine, and governed architecture.

---

## CORE PRODUCT POSITION

ICOS is not a chat system.

ICOS is a cognitive operating system.

All software must preserve:

- runtime execution
- orchestration control
- memory systems
- profile systems
- interface architecture
- voice layer
- continuity model

Any implementation that collapses ICOS into a chat interface is invalid.

---

## SYSTEM ARCHITECTURE AXIS

All software direction must align to the following axes:

### 01 · Runtime Layer

- real execution engine
- phase-based processing
- visible activity lifecycle
- no fake UI states

### 02 · Orchestration Layer

- instruction architecture
- prompt layering
- model routing
- evaluation hooks

### 03 · Memory Layer

- session memory
- profile memory
- source-linked memory
- governed persistence

### 04 · Profile Layer

- canonical intelligence surface
- voice readiness
- training depth
- reliability state
- permission states

### 05 · Interface Layer

- unified workspace shell
- no disconnected overlays
- left lane: control/navigation
- right lane: active content

### 06 · Voice Layer

- not cosmetic
- continuity-bound
- identity-sensitive
- governed activation only

### 07 · Continuity Layer

- session history as governed object
- no fake persistence
- traceable evolution

---

## UNIFIED INTELLIGENCE OPERATING SYSTEM LAYER

ICOS must operate as a full Intelligence Operating System.

The system must unify:

- cognition (thinking, planning)
- creation (documents, code, media)
- execution (runtime, deployment)
- control (computer use, automation)

All capabilities must exist inside a single governed workspace.

---

## WORKSPACE EXECUTION MODEL

Workspace is the execution surface.

System must provide:

- left lane: navigation and control
- center: active work surface
- right lane: execution state and logs
- preview layer: interactive output

All actions must occur inside the workspace.

---

## PARALLEL EXECUTION REQUIREMENT

ICOS must operate in parallel.

System must support:

- multiple active tasks
- long-running agents
- concurrent builds across domains

No serial-only execution is allowed.

---

## CREATION & DEPLOYMENT PIPELINE BINDING

All outputs must follow pipeline:

- intent → generation → refinement → validation → build → deployment

No direct output bypassing pipeline.

---

## SKILLS, PLUGINS & AUTOMATION LAYER

System must provide:

- reusable skills
- plugin extensions
- user-defined automations

Execution must be modular and composable.

---

## COMPUTER USE & DEVICE CONTROL

ICOS must be able to act on the system.

All actions must be:

- permission-bound
- visible
- interruptible

No autonomous execution without user intent.

---

## USER OPERATION MODEL

User must:

- orchestrate tasks
- operate in parallel
- control execution directly in context

Prompting is control, not conversation.

---
---

## GLOBAL SYSTEM RULES

- no local hardcoding
- no UI-only behavior masking real logic
- model selection must change real engine
- tokens must be global and shared with website
- no duplicate system logic across layers
- no overlay fixes (NOL enforced)

---

## MODEL ROUTING REQUIREMENT

Model selection must:

- switch actual runtime engine
- affect response generation
- propagate to session state
- reflect in UI indicators

UI-only switching is forbidden.

---

## SESSION SYSTEM REQUIREMENT

Session is a governed continuity object.

System must support:

- create
- save
- rename
- archive
- retrieve
- search
- temporary sessions
- model-bound sessions

Sessions must preserve runtime state and continuity boundaries.

---

## PROFILE & TRAINING REQUIREMENT

Profile is core system layer.

Must support:

- explicit user input
- voice samples
- reflection history
- thought-pattern training
- source ingestion
- external account linking

All intake must preserve:

- consent
- provenance
- weighting
- revocation

---

## PERMISSION ARCHITECTURE

System must distinguish:

- private
- public
- searchable
- interactable
- legacy-enabled

Visibility, discoverability, and interaction must never be conflated.

---

## MODEL REGISTRY REQUIREMENT

System must include governed registry:

Each model must track:

- owner
- type
- training state
- verification
- visibility
- interaction mode
- reliability

Registry must drive search and routing.

---

## DASHBOARD SYSTEM REQUIREMENT

Dashboard is core product surface.

Must include:

- model overview
- training state
- knowledge graph
- source/provenance
- reliability signals
- interaction analytics
- legacy state

No decorative analytics.

---

## SEARCH-FIRST PRINCIPLE

ICOS public experience must be search-first.

User must:

- search models immediately
- select model
- interact with real engine

No prompt-box-first paradigm.

---

## SAFETY & PRIVACY REQUIREMENT

Safety is architectural.

System must include:

- boundary enforcement
- refusal engine
- permission layers
- identity protection
- impersonation resistance
- legal review routing where required

---

## VOICE & LEGACY CONTINUITY

Voice is identity infrastructure.

Legacy continuity must:

- require explicit authorization
- remain governed
- avoid identity simulation

---

## WEBSITE ↔ SOFTWARE UNITY

Website and software must share:

- token system
- structural logic
- model registry behavior
- navigation philosophy

No divergence allowed.

---
