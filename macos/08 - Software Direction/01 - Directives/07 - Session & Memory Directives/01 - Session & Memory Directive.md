## PURPOSE

Define the canonical architecture for session handling and memory systems within ICOS.

---

## CORE POSITION

Session is a governed continuity object.

Memory is structured cognition persistence.

---

## SESSION ARCHITECTURE

System must treat each session as:

- a bounded runtime container
- linked to a model
- linked to a profile
- preserving interaction state

---

## SESSION CAPABILITIES

System must support:

- create
- save
- rename
- archive
- retrieve
- search
- temporary sessions
- model-bound sessions

---

## SESSION STATE

Each session must preserve:

- active model
- runtime state
- instruction context
- continuity boundary

---

## MEMORY LAYERS

System must distinguish:

### 01 · Session Memory

- short-term
- context-bound

### 02 · Profile Memory

- persistent
- user-linked

### 03 · Source Memory

- externally derived
- permission-bound

---

## MEMORY RULE

Memory must be:

- governed
- permission-bound
- revocable
- traceable

---

## CONTINUITY RULE

System must avoid:

- false persistence
- fake memory claims
- uncontrolled narrative carryover

Continuity must be explicit and bounded.

---

## PROFILE INTEGRATION

Memory must integrate with profile system:

- reflection history
- training signals
- pattern recognition

---

## SESSION ↔ MEMORY RELATION

Session contributes to memory.

Memory must not automatically persist without rules.

---

## SEARCH INTEGRATION

System must allow:

- session search
- memory retrieval
- context linking

---

## SOFTWARE RULE

Session and memory must exist as core system layers.

No duplication.

No UI-only state.

---

## SUCCESS CONDITION

System provides structured continuity without false identity or uncontrolled persistence.
