## PURPOSE

Define ICOS as a cognitive operating system and bind all software implementation to this identity without deviation.

---

## CORE PRODUCT POSITION

ICOS is a cognitive operating system.

ICOS is not a chatbot.

ICOS is not a conversational interface.

ICOS is a structured cognitive environment composed of interacting system layers.

---

## SYSTEM LAYERS

All software must be built as a layered system:

### 01 · Runtime Layer

- real execution engine
- stateful processing
- phase-based activity
- visible lifecycle

### 02 · Orchestration Layer

- instruction architecture
- prompt layering
- model routing
- evaluation hooks

### 03 · Memory Layer

- session memory
- profile memory
- source-linked memory
- governed persistence

### 04 · Profile Layer

- canonical intelligence surface
- training state
- voice readiness
- reliability signals

### 05 · Interface Layer

- unified workspace shell
- left lane: navigation/control
- right lane: active system surface

---

## NON-NEGOTIABLE RULES

- no feature may exist as chat-only behavior
- no UI may simulate capability without real backend state
- no model selection may remain UI-only
- no local hardcoding of system logic
- no overlay fixes (NOL enforced)

---

## RUNTIME VISIBILITY RULE

System must always expose activity state during processing.

The user must never face a blank interface.

---

## SESSION & CONTINUITY RULE

Session is a governed continuity object.

System must preserve:

- session identity
- runtime state
- model binding
- continuity boundaries

---

## PROFILE & TRAINING RULE

Profile is a core system layer.

Training is governed infrastructure, not UI enhancement.

All intake must preserve:

- consent
- provenance
- permission
- revocation

---

## MODEL ROUTING RULE

Model selection must change the actual execution engine.

UI-only switching is forbidden.

---

## PERMISSION RULE

System must distinguish:

- visibility
- discoverability
- interaction permission
- legacy activation

These must never collapse into a single state.

---

## ENFORCEMENT

Any system component that violates this directive is invalid and must be corrected at the root level.

---

## CONTINUITY RULE

All future directives and implementations must extend this directive, not bypass it.
