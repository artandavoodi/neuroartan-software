## PURPOSE

Define the canonical permission and privacy architecture for ICOS as a core system layer.

---

## CORE POSITION

Permission is system control.

Privacy is system integrity.

They are not UI settings.

---

## FUNDAMENTAL RULE

All data must be treated as sensitive by default.

Nothing is assumed public.

---

## PERMISSION DIMENSIONS

System must separate:

- visibility
- discoverability
- interaction
- training authorization
- verification authorization
- public display authorization
- legacy activation

These dimensions must never collapse into one flag.

---

## DEFAULT STATE

All user data and models must be private by default.

---

## PUBLIC STATE RULE

Public exposure must require explicit activation.

Public does not imply:

- searchable
- interactable
- trainable

Each must be independently governed.

---

## TRAINING PERMISSION RULE

Training authorization must be explicit.

System must distinguish:

- context usage
- training usage
- verification usage
- display usage

No implicit conversion is allowed.

---

## SOURCE GOVERNANCE RULE

All external sources must preserve:

- ownership
- authorization state
- intended use
- permission scope
- revocation ability

---

## IDENTITY VERIFICATION RULE

System must support:

- ownership verification
- account linking
- impersonation resistance

No source may be treated as valid without verification state.

---

## PRIVACY ENFORCEMENT RULE

System must prevent:

- unauthorized exposure
- implicit data leakage
- unintended training
- identity inference beyond consent

---

## LEGACY CONTINUITY RULE

Legacy activation must require explicit authorization.

Voice and continuity must never activate automatically.

---

## DATA REVOCATION RULE

User must be able to:

- revoke permissions
- remove sources
- disable visibility
- terminate training usage

Revocation must propagate system-wide.

---

## SYSTEM INTEGRATION

Permission must integrate with:

- model registry
- profile system
- session system
- dashboard
- search and discovery

---

## SOFTWARE RULE

Permission logic must exist as a core system layer.

No duplication.

No UI-only control.

---

## SUCCESS CONDITION

System preserves user control, identity integrity, and trust.
