## PURPOSE

Bind website and native software to one canonical system so structure, tokens, and behavior remain identical.

---

## CORE POSITION

Website and software are two surfaces of one system.

No divergence is allowed.

---

## GLOBAL TOKEN AUTHORITY

All visual and behavioral tokens must be shared:

- spacing
- typography
- color
- motion
- opacity
- component states

No local redefinition.

---

## STRUCTURAL PARITY

Both surfaces must share:

- unified workspace logic
- navigation hierarchy
- section architecture
- rail alignment
- component structure

“Same” means identical source logic.

---

## MODEL REGISTRY PARITY

Website and software must use the same registry logic:

- same data model
- same states
- same permissions
- same routing outcomes

No UI-only differences.

---

## SEARCH PARITY

Search behavior must be identical:

- same ranking rules
- same permission filtering
- same trust signals
- same results

---

## SESSION & ROUTING PARITY

Model selection must:

- bind to real engine
- persist to session
- reflect identical indicators

Across both surfaces.

---

## PERMISSION PARITY

Visibility, discoverability, and interaction must behave identically.

No website-only shortcuts.

---

## DASHBOARD PARITY

Dashboard surfaces must reflect the same data model and signals.

No decorative or simulated states.

---

## IMPLEMENTATION RULE

Fix at shared source.

Do not patch per surface.

NOL enforced.

---

## SUCCESS CONDITION

User experience remains consistent across website and native software with identical logic and signals.
