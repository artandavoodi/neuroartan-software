# ICOS Doctrine Expansion Rules

## Purpose

Defines how new doctrines, categories, and files must be created to reach full system coverage without redundancy or structural drift.

---

## Core Principle

Expansion is driven by gaps.

No doctrine is created without necessity.

---

## Gap Detection

A new doctrine is required when:

- behavior is undefined
- repeated logic appears across files
- product definition introduces new capability
- system failure reveals missing coverage

---

## Category Creation Rule

New categories must:

- follow canonical numbering
- align with existing hierarchy
- avoid duplication of scope

Categories must be:

- atomic
- clearly scoped
- non-overlapping

---

## File Creation Rule

A new file must:

- follow global metadata standard
- contain full doctrine (not placeholder)
- include injection rule
- include Change Log and Document Control

Empty files are invalid.

---

## Non-Redundancy Rule

Do not create doctrines that:

- repeat existing logic
- slightly rename existing scope
- fragment a single concept unnecessarily

Prefer extension or update over duplication.

---

## Coverage Rule

Doctrine system must achieve:

- complete behavioral coverage
- no undefined execution paths
- no conflicting rules

---

## Integration Rule

New doctrines must:

- link to related doctrines
- fit into Context Injection Order
- respect authority hierarchy

---

## Validation

Before acceptance:

- confirm gap is real
- confirm no duplication exists
- confirm alignment with product definition

---

## Enforcement

- unnecessary doctrines must be rejected
- empty or partial doctrines are invalid
- system must remain minimal and complete

---

## Injection Rule

This doctrine must be included in every prompt before inference.
