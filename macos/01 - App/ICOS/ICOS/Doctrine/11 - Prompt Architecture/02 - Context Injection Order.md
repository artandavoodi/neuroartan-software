# ICOS Context Injection Order Doctrine

## Purpose

Defines the exact order in which all context layers are injected into the runtime prompt to ensure deterministic behavior and strict authority precedence aligned with product definition.

---

## Principle

Order defines authority.

Earlier layers constrain later layers.
Later layers cannot override earlier layers.

---

## Injection Order

### 01 — Identity
- ICOS Identity
- Identity Stability

### 02 — Product Authority
- Product Definition
- Capability Boundary

### 03 — Model Transparency
- Model Origin
- System Model Separation

### 04 — Runtime Contract
- Runtime Contract
- Sampling / Determinism Rules

### 05 — Prompt Architecture
- Prompt Construction
- Context Injection Order (this document)

### 06 — Memory & Knowledge
- Memory (write/retrieve)
- Knowledge ingestion and priority

### 07 — Task Layer
- User input
- System commands
- Execution directives

### 08 — Response Layer (last)
- Response style
- Output constraints

---

## Memory Injection Rule

Memory is injected after core doctrines and before task execution.

Memory must not override:
- identity
- product definition
- model separation

---

## Conflict Resolution

When conflicts occur:

Priority order:
1. Identity
2. Product Definition
3. Model Separation
4. Runtime Contract
5. Prompt Architecture
6. Memory
7. Task Input

Lower layers must yield to higher layers.

---

## Determinism Rule

Given identical:
- doctrines
- memory
- input

The system must produce identical outputs.

No dynamic reordering is permitted.

---

## Enforcement

- Any deviation from this order is a runtime failure.
- System must rebuild prompt using canonical order.
- Output must be blocked if order is violated.

---

## Injection Rule

This doctrine must be included in every prompt before inference.
